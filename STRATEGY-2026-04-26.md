# Strategy — mission-swarm public template (2026-04-26)

Strategic-decision record for **mission-swarm** as a public template.
Mirrors the rb-047 / mission-brain pattern: public tool repo + private
user instance.

mission-swarm is currently at v0.0.0 (scaffold only); this doc locks
the public-vs-private and architectural direction NOW so a future
implementation session can build v0.1 against a settled public template
intent.

Unlike mission-brain (built but inside raybrain) or mission-bullet
(built but private), mission-swarm is pre-implementation — the
codebase is empty beyond scaffold. The strategy is mostly forward-
looking.

mission-swarm extracts the lean ~20% of MiroShark
(`github.com/aaronjmars/MiroShark`) that's useful for Ray's kriegspiel-
simulation direction. MiroShark is research-grade — Neo4j knowledge
graph, multi-platform simulation, prediction market layer, ReACT
reporting. mission-swarm keeps "personas reacting to a document, round
by round, in real time" and drops everything else.

---

## Decision 1 — Audience

**Wargame designers + kriegspiel hosts + scenario-planning teams. NOT
prediction market researchers, NOT general agent-simulation users.**

The narrow audience matters because mission-swarm's scope boundaries
are aggressive. People wanting Polymarket-style staking, knowledge-
graph-backed multi-platform simulation, or full goal-driven agent
loops should use MiroShark itself or a different tool. Mission-swarm
trades all of that for being lean, runnable on a laptop without a
GPU, and easy to invoke as a subprocess from a wargame project.

Primary use case: a wargame's GM (human or another tool) feeds a
scenario brief, gets back streamed persona reactions for the in-game
audience-reaction layer.

Secondary use cases:

- Pre-launch product reaction smoke-tests ("how does the wargame
  community react to this announcement draft?")
- Generic "how does audience X react to brief Y" sampling for
  research / strategy work
- Press-release rehearsal for sensitive announcements

GS users get a future integration story (a GS plugin surface could let
registered projects declare `uses: [mission-swarm]` and auto-invoke at
release-cycle points), but that's a decision in GeneralStaff itself,
not mission-swarm's scope.

## Decision 2 — Packaging model

**Template repo. Same call as mission-brain and mission-bullet.**

mission-swarm is opinionated about its scope but unopinionated about
its consumers. Users may want to:

- Run it as a CLI tool (the default)
- Embed it as a subprocess in a wargame engine, consuming JSON-
  streamed reactions
- Customize audience profiles in `audiences/` for their domain
- Add new persona-generation prompts for niches not covered by the
  shipped templates

Template-repo + setup-gist supports all of these without forcing a
package shape. An npm-installable form would constrain users to the
persona-generator interface as shipped, working against the
"customize for your domain" use case.

The shipped template includes:

- Core engine (persona generator, round loop, streaming output)
- Reference audience profiles (kriegspiel-domestic, kriegspiel-foreign,
  gaming-community, generic-public; possibly more)
- JSON-per-simulation persistence
- CLI + JSON-streaming-to-stdout interface

Users extend by writing additional `audiences/*.yaml` files following
the documented protocol. Forking is the supported path for new
persona-generator implementations.

## Decision 3 — Provider default (OpenRouter vs Ollama)

**Same call as mission-brain / mission-bullet: OpenRouter default,
Ollama path first-class second example. With a quality caveat.**

mission-swarm's provider needs are persona-shape generation and
reaction-text generation — both moderately demanding. Plausible
persona voice differentiation needs at minimum a 7B-class model;
small Ollama models produce visibly degraded persona reactions
(repetitive phrasing across personas, thin "stance" differentiation).

For the public README:

- **Default example**: OpenRouter + free Qwen3 Next 80B
  (currently strongest free code/reasoning model)
- **Ollama path**: documented with explicit caveat — "for full
  privacy, accept lower persona quality with small local models"
- **Claude CLI path**: documented as a quality-maximalist option for
  users with a Claude Code subscription

Privacy implication: simulation inputs (scenario briefs, press
releases) may be sensitive. The Ollama path is the airtight option
for sensitive content; the OpenRouter path is for non-sensitive
simulations or users comfortable with OpenRouter's data handling.

## Decision 4 — Privacy story

**Inputs may be sensitive. Defaults: simulations/ gitignored, no auto-
upload, opt-in to cloud providers.**

Unlike mission-brain (corpus is the user's writing) or mission-bullet
(entries are the user's daily journal), mission-swarm's inputs are
scenario briefs that may include:

- Pre-release product information (announcement drafts, pivot copy)
- Strategic scenarios (wargame briefings, internal communication
  drafts)
- Sensitive content the user fed it that shouldn't become public

README hero target (subject to Ray voice-pass before publication):

> Your scenario briefs stay on disk. mission-swarm uploads input only
> when you've configured a cloud provider, and the simulations/ output
> directory is gitignored by default. The Ollama path is fully local —
> your inputs never leave your machine.

simulations/ gitignore is load-bearing. Audience-profile templates in
audiences/ ARE tracked because they're reusable shape definitions, not
the user's content.

## Decision 5 — Positioning vs alternatives

**Distinctive pitch: lean kriegspiel reactions + audience-template-
driven + zero-infra.**

| Tool | What it does | What mission-swarm does differently |
|---|---|---|
| **MiroShark** | Research-grade knowledge-graph + multi-platform simulation | The lean ~20% useful for kriegspiel; no Neo4j, no GPU required |
| **ChatGPT-as-personas** | Ad-hoc persona prompting in a chat session | Structured rounds, persona persistence across rounds, audience templates, streamable JSON output |
| **SimsForge / general agent sim** | Goal-driven autonomous agents | Reactions-to-input, not goal-driven; scope is intentionally limited |
| **Polymarket / prediction markets** | Real staking + market dynamics | Simulation only, no AMM, no real money |

The four differentiators that stay in every public surface:

1. **Lean.** Runnable on a laptop without a GPU. Bun + TypeScript +
   JSON; no databases, no graph layer, no message queues.
2. **Reactive, not goal-driven.** Personas respond to your input
   document; they don't pursue objectives. Scope is bounded by intent.
3. **Audience-template-driven.** Plug in
   `audiences/wargame-german-1900s.yaml` for plausible Prussian-officer
   personas; swap to `audiences/anglosphere-tech.yaml` for HN
   reactions. Templates are the customization point.
4. **Streamable.** Reactions arrive round-by-round in real time,
   consumable by parent simulations as they generate (not a
   batch-and-dump tool).

## Phase 2 — implementation plan (for a fresh session)

mission-swarm is at v0.0.0 (scaffold only). Phase 2 is "build v0.1
against this strategy" rather than "publish what exists." Expect
~10-12 hours of focused engineering across multiple sessions.

### Step 1 — Persona generator (~3-4 hours, fresh session)

Build the LLM-driven persona generator:

- Input: audience profile YAML + N (count of personas)
- Output: array of `{name, bio, stance, interests, style}` persona
  objects
- Provider abstraction: OpenRouter default, Ollama path, Claude CLI
  optional

Reference: MiroShark's persona generation has the basic shape;
mission-swarm's is simpler (no belief-state tracking, no graph
context fetch).

### Step 2 — Round loop (~3-4 hours)

Build the round-based reaction loop:

- For each round R in [1..N]:
  - For each persona P:
    - Construct prompt: P's persona + recent rounds' feed + input doc
    - LLM call → reaction text + updated stance/interest
    - Stream reaction to stdout (newline-delimited JSON)
- Persist round state to `simulations/<id>/round-N.json`

Streaming is load-bearing for "real-time consumable by parent
simulations." Don't batch-and-dump.

### Step 3 — Audience profile templates (~2 hours, voice-bearing)

Ship 4-6 reference audience profiles in `audiences/`:

- `kriegspiel-domestic-1900s.yaml` — Prussian/German officer corps,
  civilian press, parliamentary
- `kriegspiel-foreign-1900s.yaml` — British, French, Russian observers
- `gaming-community.yaml` — HN, indie devs, wargame forum demographics
- `pixel-art-community.yaml` — digital artists, retro-game fans
- `generic-public.yaml` — untargeted demographic mix
- One more yet to choose (possibly `wargame-publisher-pros.yaml` for
  industry-insider reactions)

Ray's pen on the actual persona-generation prompts in each profile;
templates are voice-bearing.

### Step 4 — CLI + JSON streaming (~1-2 hours)

`bun src/index.ts <input-doc> --audience <name> --rounds N --personas
K`. Streams reactions as JSON-per-line to stdout. Optional SSE flag
for HTTP consumers.

### Step 5 — Public-readiness pass (~30 min)

Same as mission-bullet Step 1 — scrub for Ray-personal references,
generalize examples, README final voice pass with stop-slop.

### Step 6 — `gh repo edit lerugray/mission-swarm --visibility public`

Repo flips public after Steps 1-5 are green AND v0.1 demonstrably runs
end-to-end on a fresh machine.

### Step 7 — Wire into public GS README

Add mission-swarm to the public GS README ecosystem section
(coordinated with mission-brain and mission-bullet additions for a
single coherent commit).

## Risk + open items

- **Persona quality on Ollama is hand-wavy.** mission-swarm's
  reactions need plausible voice differentiation; Ollama with 7B-class
  models may not deliver. Mitigation: README is honest about it,
  default example uses OpenRouter, Ollama path is documented as the
  privacy-tradeoff option.

- **Audience-template authoring is voice-bearing.** Each shipped
  audience profile has prompts that shape persona generation; getting
  them right takes Ray's pen + iteration. Templates are user-
  replaceable, but the shipped defaults set the quality bar.

- **MiroShark attribution.** mission-swarm explicitly extracts ~20% of
  MiroShark's design; README must credit `aaronjmars`'s repo upfront,
  with a clear "what's the same, what's lean'd out" framing.

- **Public push gated on Phase 2 Steps 1-5 actually working.** Until
  v0.1 demonstrably runs end-to-end, public push is premature. Unlike
  mission-brain and mission-bullet (already-built tools), mission-
  swarm has nothing to publish yet.

- **Naming collision.** `mission-swarm` namespace — npm/PyPI/GitHub-
  org-name check before public push.

- **GS plugin surface design.** If/when GS gains a plugin layer for
  registered projects to declare `uses: [mission-swarm]`, that's a
  separate gs-XXX task — keep mission-swarm's design free of
  GS-specific assumptions so the plugin layer can wrap rather than
  cross-cut.
